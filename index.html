The 2 AM NOC Incident Problem

You are a DevOps engineer at a company called NovaCloud. Several Ubuntu VPS servers and the services running on them are going down for different reasons. Monitoring is noisy, alerts are piling up, and your task is to build a Python program that can inspect a large nested dictionary, filter useful information, identify what is broken, and decide what actions are valid.

This is a problem statement only. Do not solve it directly. Build your own Python solution using the concepts you already know: dictionaries, tuples, loops, conditions, lists, and user input.

Infrastructure snapshot

You receive a single dictionary called infrastructure. Each key is a server ID, and each value is another dictionary containing metadata about the server and its services.

Your code should be designed so you can inspect the whole infrastructure, but also narrow it down with filters like region, team, tag, server status, or service status.

Your tasks

1

Basic inventory

Print a summary of every server: server ID, hostname, region, datacenter, and current server status. Make it readable and clean, one server per line.

2

Region filter

Ask the user to enter a region such as eu-central, eu-west, us-east, or ap-southeast. Show only the matching servers, plus their status and hostnames.

3

Find all down services

Loop through every server and every service. Collect all services whose status is down. For each one, display the server ID, hostname, service name, last error, and which team owns that server.

4

Restart decision logic

For each down service, determine whether it can be restarted or not. Do not actually restart anything in this task. Just decide whether restart is allowed and print the reason clearly.

  • Cannot restart if the server itself is down.
  • Cannot restart if auto_restart is False.
  • Cannot restart if restart_attempts is equal to max_restart_attempts.
  • Cannot restart if max_restart_attempts is 0.
  • Cannot restart if any service listed in depends_on is also down on the same server.
  • Can restart only if none of the blockers above exist.
5

Resource danger zones

Find all servers where at least one resource is in a dangerous range.

  • CPU above 85%
  • RAM above 85%
  • Disk above 90%

Your output should say exactly what resource is dangerous on each matching server.

6

Team alert report

Group affected servers by on_call_team. Only include servers that have at least one down service. Show which team should be alerted and which servers belong to that team.

7

Tag filter

Ask the user for a tag such as production, staging, database, or web. Print all servers that include that tag, together with the number of down services on each server.

8

Action script challenge

Create a separate script that tries to turn on services after checking them. This script must first inspect each service and decide whether an action is valid based on the operational rules. If a service is down and all restart conditions are satisfied, the script should attempt to mark it as started. If a service is down but cannot be started, the script should skip it and clearly print the reason.

Important: this task is part of the problem statement. Do not forget it. The script should never try to start services blindly.

Operational rules

Rule 1: If the server is down, you cannot act on its services at all.
Rule 2: A service with a dependency cannot start until all required dependencies on the same server are online.
Rule 3: If auto_restart is false, the service requires manual intervention.
Rule 4: If max_restart_attempts is 0, the service is not configured for restart attempts.
Rule 5: If restart attempts already reached the maximum, no more attempts are allowed.
Rule 6: A degraded server is reachable but unstable. Treat it as a warning, not as a full blocker.
Rule 7: If port is None, that does not automatically mean the service is broken. It may be a worker or daemon.
Rule 8: Very high disk usage may explain storage errors such as no space left on device.
Rule 9: Very high RAM usage may help explain memory-related service failures such as OOM errors.
Rule 10: One server can have multiple different problems at the same time. Your logic should not stop at the first issue.

Creative extras

Extra filters you can build

  • Filter only production servers in eu-central.
  • Filter only servers owned by team-alpha.
  • Filter only servers with status degraded.
  • Filter only services with dependency problems.
  • Filter only services failing because of storage or memory symptoms.
  • Filter only background services where port is None.

Hard-mode ideas

  • Create a menu so the user can choose which report to run.
  • Store findings in new dictionaries or tuples for later reporting.
  • Sort outputs by region, server status, or number of failures.
  • Add a maintenance mode tag and treat it differently from outages.
  • Add a timestamp field later and report the newest failures first.
  • Create a final incident summary for each affected server.
